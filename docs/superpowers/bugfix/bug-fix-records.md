## 1. 大模型幻觉：输出完整简历而非回答问题

### 现象

用户上传简历后提问"花小猪砍车费是什么"，AI 模型不是针对性地回答问题，
而是将**整份简历内容全部输出**，夹杂了大量不相关的个人信息和幻觉内容。

### 根因分析

**1. 智谱 GLM-4V 的系统指令跟随能力弱**

系统提示词设置"答案必须来自资料原文，禁止添加资料中没有的信息"，
但 GLM-4V 对 System Message 的响应程度远低于 User Message。
模型倾向于忽略系统层的约束，优先遵循用户消息中的指令。

**2. 文件内容拼接方式不当**

用户上传的文件内容直接拼接在用户消息后面，模型将大段文件内容理解为
"需要处理的输入"而不是"可以参考的资料"，导致模型认为用户要求它复述
或处理整份文件。文件内容的呈现方式模糊了"参考材料"和"待处理输入"的边界。

**3. 模型预训练知识干扰**

针对"花小猪"这类知名度高的业务，模型有很强的预训练记忆。这些记忆与
用户的简历内容混合后，模型产生了幻觉，补充了简历中没有的信息。

### 解决方案

多模型路由 + 指令内嵌：

```java
// 1. 路由策略：纯文本走 DeepSeek，图片才走智谱
if (hasImages) {
    streamChatWithZhipu(...);
} else {
    streamChatWithDeepSeek(...);
}

// 2. 指令内嵌到用户消息中（对两种模型都有效）
String enriched = "请分析以下资料来回答问题。\n"
    + "要求：答案必须直接引用资料中的原文，不得添加资料中没有的信息。\n"
    + "\n文件内容：\n" + fileContext
    + "\n问题：" + userMessage;
```

## 2. RAG 无条件注入导致模型无法回答通用问题

### 现象

用户提问"你好"或"Python requests 库怎么用"等与知识库无关的问题时，
模型回答"没有找到相关信息"，拒绝回答本可以回答的通用问题。

### 根因分析

**RAG 架构缺乏意图识别层**

原流程：`用户提问 → 向量检索 → 无条件注入上下文 → 强制引用 → 输出`

三个问题叠加：
1. **没有相关性判断**：检索到的内容不管和问题有没有关系，一律注入
2. **系统提示词过于严格**："禁止添加资料中没有的信息"导致模型被禁止使用自身知识
3. **检索即假设**：错误地假设 TOP-K 结果一定和用户问题相关

### 解决方案

余弦相似度阈值过滤 + 双模式 Prompt：

```java
// 1. 相关性阈值（经实际数据校准为 0.55）
private static final double RELEVANCE_THRESHOLD = 0.55;

// 2. 阈值过滤
double bestScore = scoredVectors.get(0).similarity();
if (bestScore < RELEVANCE_THRESHOLD) {
    return ""; // 低于阈值：不注入 RAG 上下文
}

// 3. 双模式 Prompt
// 有上下文时 → "请分析以下资料来回答问题。答案必须直接引用资料中的原文..."
// 无上下文时 → 系统提示词"你是智能助手" + 直接走用户消息
```

**阈值校准**：ZhiPu embedding-2 模型对中文相似度分布偏低。
相关查询 0.58~0.62，不相关查询 0.25~0.33，校准为 0.55。
不引入额外 API 做意图分类，余弦距离本身就是天然的相关性信号。

## 3. 智谱 GLM-4V 忽略系统指令

### 现象

系统提示词设置"答案必须来自资料原文"，但 GLM-4V 仍然基于自己的预训练知识回答，
甚至在资料中不包含相关信息时编造内容。

### 根因分析

1. **System Message 权重低**：智谱 API 设计将系统消息的优先级置于用户消息之下
2. **多模态模型特性**：GLM-4V 优化方向是图片理解，对文本指令的精细控制不如纯文本模型
3. **架构差异**：OpenAI 兼容 API 中 System Role 在不同模型上的实现权重不同

### 解决方案

指令内嵌（Instruction Injection）——将约束写入 User Message：

```java
String enriched = "请分析以下资料来回答问题。\n"
    + "要求：答案必须直接引用资料中的原文，不得添加资料中没有的信息。\n"
    + "\n文件内容：\n" + fileContext
    + "\n问题：" + userMessage;
```

配合多模型路由：纯文本 → DeepSeek，包含图片 → GLM-4V（消息中已内嵌约束）。

## 4. 面试话术

### Q1：RAG 系统的意图识别你是怎么设计的？

> "我们没有用额外的意图分类器。核心思路是：**向量检索的余弦相似度本身就是
> 最好的相关性信号**。我们给检索结果设了一个 0.55 的相似度阈值，低于这个值
> 就不注入 RAG 上下文，让模型用自己的知识回答。同时实现了**双模式 Prompt**：
> 有上下文时要求严格引用来源，无上下文时允许自由回答。这个方案的优点是
> 零额外成本（不需要额外的 API 调用），利用 embedding 模型已有的余弦距离
> 做判断。阈值是根据实际 embedding 模型（ZhiPu embedding-2）的数据分布
> 校准的——相关查询相似度 0.58 左右，不相关的 0.3 以下，0.55 是平衡点。"

### Q2：你怎么解决大模型幻觉问题的？

> "我们遇到了两类幻觉：一是模型无视系统指令自行编造，二是 RAG 注入
> 不相关内容后模型强行引用。解决方案分三层：
>
> 第一层，**多模型路由**——文本对话用 DeepSeek，多模态（图片）才用
> GLM-4V，因为 DeepSeek 的指令跟随能力明显更强，引用也更规范。
>
> 第二层，**指令内嵌**——关键约束同时写在系统提示词和用户消息中。
> 因为不同模型对 System Message 的权重不同，写到 User Message 里
> 对所有人都有效，这是通用最佳实践。
>
> 第三层，**相关性过滤**——通过余弦相似度阈值只注入真正相关的内容，
> 避免模型被不相关的检索结果误导。"

### Q3：你遇到过什么和 RAG 相关的有意思的问题？

> "有一个很有意思的问题：embedding 模型的相似度分布差异很大。
> 我们一开始用的 OpenAI ada-002，相似度 0.8 以上的才算相关。
> 后来切到 ZhiPu embedding-2，发现最相关的查询也只有 0.62，
> 0.6 的阈值把很多正确答案过滤掉了。最后降到了 0.55。
>
> 这个教训是：**RAG 系统的阈值不能拍脑袋，必须基于实际使用的
> embedding 模型来校准**。每个模型的向量空间分布不一样，
> 不同语言（中文 vs 英文）的语义空间也不一样。
> 做 RAG 系统一定要有可视化和调优的过程。"
